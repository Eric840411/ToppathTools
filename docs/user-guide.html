系統總覽

Toppath Tools 是一套 QA、PM、OSM 維運與 Game Show 測試工具集合，前端以 tab 分群呈現，後端統一由 Express 提供 API、SSE 與 WebSocket。

前端功能分群

群組工具主要用途
JiraJira 批次建立從 Lark Sheet、Google Sheet 或 Lark Bitable 讀資料，批次建立 Jira Issue、補留言、轉狀態。
TestCaseTestCase 生成讀取需求 URL 或文件，透過 AI 產生測試案例，輸出或寫回 Lark。
OSM ToolsOSM Version、Machine Test、Image Check、Config Compare、AutoSpin、URL Pool、Jackpot、OSM UAT處理 OSM 版本同步、機台自動測試、圖片比對、設定比對、長時間 Spin、共用 URL、Jackpot 監控與後台 UAT。
Game ShowPDF TestCase、圖片比對、500x 統計、Log Checker針對 Game Show 需求文件、畫面、WebSocket 統計與 log 檢查做輔助測試。
History歷史紀錄查詢工具操作紀錄、執行結果與錯誤資訊。
全域設定Jira 帳號、AI Key、Prompt、Agent Monitor管理身分、AI 模型、API Key、Prompt 範本與 Agent 監控。

啟動、建置與部署流程

依使用場景選擇開發模式、內部開發模式或 production tunnel 模式。

一般開發模式

npm run dev:all
  1. 先執行 dev:stop，清掉 3000、5173 的殘留 process。
  2. 啟動 tsx --watch server/index.ts，API 監聽 0.0.0.0:3000。
  3. 啟動 vite，前端監聽 0.0.0.0:5173。
  4. 區網使用者可透過 http://192.168.x.x:5173 存取前端。

內部開發模式

npm run dev:internal
  1. 清掉 3001、5174。
  2. 後端由 server/index.dev.ts 設定 PORT=3001。
  3. 前端使用 vite.dev.config.ts，API/WS proxy 到 3001。

Production Build + Tunnel

雙擊 build-and-tunnel.bat
或執行 powershell -ExecutionPolicy Bypass -File build-and-tunnel.ps1
  1. 停止 3000、5173。
  2. 執行 npm run build 產出 dist。
  3. 啟動 npx tsx server/index.ts，Express 會同時提供 API 與 production 前端。
  4. 啟動 Cloudflare Tunnel 指向 http://localhost:3000。
  5. 自動抓出 trycloudflare.com URL，組成 OSMWatcher Webhook 並複製到剪貼簿。

Jira 批次建立與流程管理

支援 QA 與 PM 兩種工作流，核心是把 Lark/Google 資料整理成 Jira Issue，並回寫 key、補留言與轉狀態。

應用場景

  • QA 從 Lark Sheet 或 Google Sheet 匯入 Issue 清單，批次建立 Jira ticket。
  • PM 從 Lark Bitable 讀需求，依 Parent Epic 建立 Story 或相關 Issue。
  • 建立完成後回寫 Jira Issue Key，後續可批次補上測試結果留言、附檔連結、AI 產生的說明與 verifier。
  • 依 Jira workflow transition id 批次轉狀態。

主要流程

1. 選帳號設定 Jira email、API token、角色與 PIN。
2. 讀資料貼上 Lark Sheet、Google Sheet 或 Bitable URL。
3. 預覽校對檢查 summary、description、assignee、issue type、project。
4. 批次建立呼叫 Jira REST API 建立 issue 並整理成功/失敗結果。
5. 回寫將 Jira key 與狀態回寫到 Sheet/Bitable。
6. 補留言批次將測試結果、檔案、AI 內容貼到 Jira comment。
7. 轉狀態依 transition id 批次推進 Jira workflow。
8. 留紀錄操作寫入 history，方便查詢。

輸入與結果

項目內容
必要輸入Jira 帳號、API Token、Project、Issue Type、資料來源 URL。
常見欄位Summary、Description、Assignee Email、RD、Verifier、Difficulty、Jira Issue Key、Result。
成功結果回傳 issue key、Jira browse link、建立狀態；可回寫來源表格。
失敗結果保留 Jira API 錯誤、欄位缺漏、權限問題或 assignee mapping 失敗原因。

角色與資料流

QA 模式從 Lark Sheet / Google Sheet 讀入測試或缺陷清單，建立 Jira issue 後回寫 Jira Key，再用批次留言補上測試結果、附件連結與 AI 生成的說明。
PM 模式從 Lark Bitable 讀取需求結構，帶入 Parent Epic Key，建立 Story 或 Epic 相關 issue，適合需求拆分與 Jira 初始化。
安全控管Jira 帳號支援角色、PIN 驗證與 admin 驗證；敏感 token 不顯示在前端結果，只保留必要操作紀錄。

TestCase 生成

把需求文件、Lark 內容或檔案交給 AI，產出可回填到 Lark Bitable/Sheet 的測試案例。

應用場景

  • 需求文件太長，需要快速整理測試點、前置條件、步驟與預期結果。
  • PM/QA 希望從 Lark URL 直接生成 TestCase，不手動複製內容。
  • 需要用同一套 Prompt 模板維持輸出格式一致。

主要流程

1. 選模型選 Gemini、OpenAI 或可用模型。
2. 輸入來源Lark URL、需求文字、PDF/Word 等文件。
3. AI 解析依 Prompt 產生測項、步驟、預期結果。
4. 預覽校正確認欄位與語意。
5. 輸出下載檔案或寫回 Lark。
6. 監控SSE 顯示生成進度。

結果格式

欄位說明
測試分類功能、UI、資料、錯誤處理、邊界條件等。
測試標題簡短描述測試目的。
前置條件帳號、環境、資料、遊戲狀態。
測試步驟可執行的逐步操作。
預期結果判定 PASS/FAIL 的結果描述。
優先級/備註補充風險、來源章節與特殊條件。

OSM Version Sync 與版本監控

集中管理 OSM、LuckyLink、Toppath 與 ImageRecon 版本資訊，並可同步到 Lark。

應用場景

  • 查詢 CP、WF、TBR、TBP、MDR 等 channel 的 backend/frontend 版本。
  • 同步 OSM 版本資料到 Lark 表格。
  • 抓 LuckyLink、Toppath 版本歷史並整併成報表。
  • 讀 Gmail 中 ImageRecon 週報，彙整 server 狀態。
  • 管理目標機型與設定模板，用於後續比對或測試。

主要流程

1. 讀取 channels從環境變數與 route 取得 channel 設定。
2. 查版本呼叫 OSM/LuckyLink/Toppath backend。
3. 同步 Lark依設定寫入 Bitable/Sheet。
4. 顯示報表前端呈現各 channel 狀態與時間。

結果

  • OSM version history：每個 channel 的 GM/API/GW 等版本資訊。
  • LuckyLink/Toppath history：對應服務版本與同步時間。
  • ImageRecon 報表：從 Gmail 抓取最新報告並彙整狀態。
  • 同步結果：成功筆數、失敗筆數、錯誤訊息。

機台自動測試

以 Playwright、自動操作、截圖、音訊與 OSMWatcher 狀態輔助，對機台 URL 執行登入、Spin、Bonus、iDeck、CCTV 等測試。

應用場景

  • 批次測試多台機台，並將 PASS/FAIL/WARN 結果寫回 Lark。
  • 測試遊戲登入、Spin、Bonus、音訊、iDeck、CCTV、錯誤畫面。
  • 分散式 agent 由多台機器共同 claim job，加速大量機台測試。
  • OSMWatcher 透過 webhook 回報機台狀態，測試 runner 可依狀態判斷等待或繼續。

主要流程

1. 輸入 URL設定 worker/lobby URL 與 token。
2. 讀 Lark取得機台清單、機型、預期結果。
3. 選 Profile套用機型觸控點、selector、bonus 動作。
4. 開始測試本機或分散式 agent 執行。
5. 即時串流WebSocket 推送 log、截圖、queue 狀態。
6. 寫回結果測試完成後寫回 Lark 並保存歷史。

OSM UAT 後台測試

從 Lark Bitable 掃描 TestCase，自動執行 OSM 後台 UAT 測試並串流進度。

應用場景

  • 根據 Lark Bitable 的 TC 清單執行後台功能驗證。
  • 涵蓋排序、Bonus、Export、Config 等常見後台測試。
  • 以 SSE 即時顯示執行進度、錯誤、停止狀態。

主要流程

1. 掃描 TC提供 Lark URL，讀取可執行測項。
2. 選擇測項確認要跑的測試類型與環境。
3. 執行後端 runner 執行測試。
4. 串流結果SSE 回傳 log、PASS/FAIL 與停止狀態。

Agent 控制面

下載與安裝

前端可從後端下載 install.bat、launcher.bat、agent.py，讓測試機快速接入中央 server。

遠端操作

Server 可對 agent 下達 start、stop、pause、resume、spin interval 等指令，也會接收 screenshot 與 log。

模板與辨識

模板圖存放於 server/python/templates，AutoSpin 透過 OpenCV 比對畫面狀態、按鈕與錯誤提示。

對帳報表

reconcile 會整理 agent 回傳的 game-record 與設定資料，產出可查詢的 reports。

圖片檢查

針對改圖、素材或遊戲畫面做自動截圖與 AI/人工檢查，判斷圖片是否正確載入或出現異常。

應用場景

  • RD 提供修改過的圖片清單，QA 需要快速檢查是否進入遊戲。
  • 貼上遊戲 URL 與圖片 path，工具自動開啟頁面、截圖、標記檢查結果。
  • 支援手動 click、scroll、停止 session、查看截圖。

主要流程

1. 輸入 URL遊戲或入口 URL，通常含 token/gameid。
2. 輸入圖片貼上 git diff、圖片 path 或 AI 整理後的清單。
3. 開始檢查後端啟動 browser session。
4. 擷取畫面取得 screenshot 並解析。
5. 互動確認必要時 click/scroll 後重抓畫面。
6. 輸出結果列出已找到、疑似缺漏、需要人工確認的項目。

AutoSpin

使用 Python agent、OpenCV、模板辨識與遠端控制，執行長時間 Spin、紀錄與對帳。

應用場景

  • 長時間跑 spin，觀察遊戲是否卡住或進入特殊狀態。
  • 使用模板圖辨識按鈕、錯誤、bonus、彈窗等畫面。
  • 支援下載 agent、遠端上傳 screenshot/log、暫停、恢復、停止。
  • 提供 reconcile 功能，比對遊戲紀錄與設定。

主要流程

1. 設定 config機型、URL、動作、模板與 bet random。
2. 啟動 session後端呼叫 Python 或等待 agent 連線。
3. 執行動作依 actions.json 點擊、等待、辨識畫面。
4. 串流 logSSE 顯示 session 或 agent log。
5. 保存結果history、game-record、截圖與 reconcile report。
