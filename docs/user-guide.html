J

自動寫入 Jira

從 Lark Sheet / Google Sheet 資料批次建立 Issue、新增評論、切換處理狀態

功能說明

此功能支援兩種操作模式，請依據您的職位選擇：

🧪 QA 模式

從試算表讀取測試資料，批次建立 Jira Issue、寫入評論、切換狀態

  • 來源：Lark Sheet 或 Google Sheet
  • 欄位：摘要、內容、受託人、RD負責人、驗證人員等
  • 支援批次評論 + AI 生成評論
  • 支援 Lark Drive 附件自動上傳至 Jira

📋 PM 模式

從 Lark 多維表格（Bitable）讀取需求清單，批次建立 Jira Story

  • 來源：Lark Bitable URL
  • 可設定 Parent Epic Key
  • 支援選擇子集建立

Step 1 — 設定 Jira 帳號

  • 點擊右上角「帳號設定」或畫面頂端的帳號選擇區
  • 輸入 Jira 登入 Email 及 API Token（從 Atlassian 帳號管理頁取得）
  • 設定帳號角色（QA / PM），決定後續可使用的模式
  • 帳號會儲存於本次 Session，重新整理後需重新選擇
ℹ️
API Token 取得路徑：Atlassian 帳號設定 → 安全性 → API Token → 建立新 Token

Step 2 — 選擇受託人 / 專案 / Issue 類型

  • 系統自動載入您有權限的 Jira 專案列表
  • 選擇目標專案後，系統自動載入該專案的 Issue 類型與成員
  • 從下拉選單選擇受託人（Assignee）

Step 3 — 匯入試算表

  • 選擇來源類型：Lark Sheet 或 Google Sheet
  • 貼上試算表網址（Lark 格式：https://casinoplus.sg.larksuite.com/sheets/XXXX）
  • 點擊「讀取資料」，系統自動抓取所有資料列
⚠️
Lark Sheet 需將機器人加入文件成員（分享 → 加入 App），否則會回傳權限錯誤
ℹ️
試算表欄位對應：「摘要」→ Summary、「內容」→ Description、「受託人」→ Assignee Email 等。欄位名稱不分大小寫。

Step 4 — 確認清單 / 勾選資料列

  • 讀取後顯示所有資料列，狀態標籤說明：
    • 待開單 — 尚無 Jira Key，需建立 Issue
    • 已開單 — 已有 Jira Key，可進行評論步驟
    • 已評論 — 評論完成，可切換狀態
  • 可全選或個別勾選要操作的列

Step 5 — 建立 Issues

  • 點擊「建立 Issues」，系統批次提交至 Jira
  • 成功後，試算表「jira issue key」欄位自動回寫 Issue Key
  • 建立失敗的列會顯示錯誤訊息，可單獨重試

Step 6 — 添加評論

  • 選擇評論內容來源的欄位（試算表中的欄位名稱）
  • 可選擇「附件欄位」：若該欄位有 Lark Drive 連結，系統自動下載並上傳至 Jira
  • 可開啟「AI 生成評論」，選擇 Gemini Prompt 模板，由 AI 依資料撰寫評論
  • 點擊「批次添加評論」完成

Step 7 — 切換狀態

  • 系統自動找出「處理階段 = 已評論」的 Issues
  • 點擊「切換狀態」，批次將 Issue 推進至下一個 Workflow 狀態
  • 成功後試算表「處理階段」欄位自動更新

TestCase 生成

輸入規格書，由 Gemini AI 自動生成測試案例並寫入 Lark 多維表格

兩種操作模式

🤖 AI 生成 TestCase

輸入規格書，AI 分析後自動產生完整測試案例清單，寫入 Lark Bitable

🔍 人工 vs AI 比對

同時輸入規格書 + 你已寫好的 TestCase Bitable，AI 分析覆蓋率與缺漏

4
（選填）Jira 整合
→
5
開始生成
→
6
查看結果 / Bitable

Step 1 — 選擇規格書來源

  • 📄 Lark Wiki — 貼上 Lark Wiki 文件網址
  • 📎 PDF / Word — 上傳 .pdf 或 .docx 規格書（最大 20MB）
  • 🌐 Google 文檔 — 貼上 Google Docs 連結（文件需設為「任何人可查看」）

Step 2 — 輸入規格書網址或上傳檔案

  • Lark Wiki：系統自動從 URL 抽取 /wiki/{documentId}
  • PDF/Word：點擊「選擇檔案」選取本機檔案
  • Google Docs：貼上完整分享連結
⚠️
Lark Wiki 需先在文件「分享」中加入 Lark Bot，否則系統無法讀取文件內容

Step 3 — 選擇 AI Prompt 模板

  • 從下拉選單選擇適合的生成模板
  • 標準格式：生成包含「測試模組、維度、標題、前置條件、步驟、結果、優先級」的欄位
  • Jira 整合格式：生成包含「功能模組、測試類型、來源依據、Jira 單號」等欄位
  • Prompt 模板可在 ⚙️ Gemini 設定中自訂與新增

Step 4 — Jira 整合（選填）

  • 勾選「整合 Jira 單號」，讓 AI 同時參考 Jira Issues 的描述
  • 選擇 Jira 帳號，輸入相關 Jira 單號（逗號或換行分隔，如 CGSG-220, CGSG-221）
  • AI 會將 Jira Issues 內容納入分析，生成更精確的測試案例

Step 5 — 開始生成（約 30–60 秒）

  • 點擊「開始生成」，系統進入 Loading 狀態
  • 後端流程：讀取文件全文 → 送至 Gemini 分析 → 批次寫入 Lark Bitable
  • 生成期間請勿離開頁面

Step 6 — 查看結果

  • 頁面顯示生成筆數、寫入筆數摘要
  • 點擊「前往 Lark Bitable 查看結果」開啟已建立的多維表格
  • 下方預覽表格可展開查看每筆案例的詳細欄位

O

OSM 版號同步

即時同步各渠道機台版本，與 Lark 目標版本比對，掌握全局版本狀態

功能說明

  • 從多個 OSM 渠道（CP / WF / TBR / TBP / MDR）拉取所有機台的當前版本
  • 與 Lark Sheet 設定的「目標版本」比對，標示是否達標
  • 顯示版本分布圖、在線 / 離線統計
  • 可解析 ImageRecon 週報 Email

主要操作區塊

目標版本設定

  • 貼上 Lark Sheet 網址（格式：https://casinoplus.sg.larksuite.com/sheets/XXXX）
  • 點擊「從 Lark 同步」，自動讀取各渠道目標版本
  • 可手動設定告警排程，當版本不符時推播到 Lark 群組

機台版本 Dashboard

  • 點擊「全部同步」，從所有渠道拉取最新機台版本資訊
  • 依渠道分組顯示，每組有版本分布色條圖
  • 機台狀態：online / offline / unknown

OSM 元件版本

  • 點擊「取得元件版本」，從 OSM 後台拉取各元件（GM / API / GW 等）的版本
  • 與 Lark Sheet 目標版本比對，顯示是否符合要求

LuckyLink / Toppath 元件版本

  • 分別從 LuckyLink 後台 / Toppath 靜態接口拉取元件版本
  • 同樣與 Lark 目標版本比對

ImageRecon Server 版本

  • 點擊「取得最新報告」，從最新一封 Gmail 週報 Email 解析 Server 狀態
  • 與 Lark Sheet ImageRecon 目標版本比對

機台自動化測試

批量自動測試機台的進入、推流、Spin、音頻等流程，結果自動回寫 Lark

1

→
2

→
3
輸入機台清單
→
4
選擇測試項目
→
5
開始測試
→
6
查看結果

Step 1 — 設定大廳 URL（Worker）

  • 每個 Worker 對應一組大廳 URL（含 token）
  • 可新增多個 Worker，系統會平行分配機台測試
  • 格式範例：https://example.com/lobby?token=ABC123
  • 每個 Worker 使用不同的帳號 token，各自獨立測試不同機台

Step 2 — 設定 Lark 測試表單（選填）

  • 貼上 Lark Sheet URL，系統會自動讀取機台代碼清單
  • 測試完成後，結果（PASS / FAIL）自動回寫至表單對應列

Step 3 — 輸入機台代碼清單

  • 每行一個機台代碼（或用逗號分隔），如 892-BULLBLITZ-0136
  • 也可點擊「讀取機台」從 Lark 表單自動匯入

Step 4 — 選擇測試項目

測試項目說明狀態
進入機台從大廳導航至指定機台並確認進入成功可用
推流檢測確認串流畫面正常接收可用
Spin 測試執行指定次數的 Spin，記錄結果可用
音頻檢測偵測遊戲音效是否正常播放（需 VB-Cable）可用
iDeck 測試測試觸屏機台的 iDeck 互動介面開發中
退出測試確認可正常退出機台回到大廳可用

Step 5 — 開始測試

  • 點擊「開始測試」，系統依機台清單逐一（或平行）執行測試
  • 左側即時日誌顯示每台的執行進度
  • 日誌中可確認 OSMWatcher 圖像識別服務是否連線
ℹ️
若 OSMWatcher 已連線，測試時 Spin 後會自動偵測特殊遊戲（Free Game / Jackpot / Handpay），等待結束後再繼續下一台
⚠️
若 OSMWatcher 未連線，Spin 後不會等待特殊遊戲結束，日誌會標示「⚠️ OSMWatcher 未連線」

Step 6 — 查看結果

  • 右側「測試摘要」顯示每台機台的整體 PASS / FAIL / WARN
  • 可展開查看每個測試步驟的詳細結果與耗時
  • 若設定了 Lark 表單，結果同步回寫

圖片刪除驗證

貼上已刪除圖片清單，自動在虛擬瀏覽器中驗證是否仍被遊戲載入

使用情境

💡
當 RD 刪除遊戲圖片資源後，QA 需確認圖片真的沒有被前端載入（而非只是 Git 刪除成功）。此工具自動化完成此驗證流程。

操作流程

1
輸入前端 URL
→
2
填入刪除圖片清單
→
3
啟動偵測
→
4
操作虛擬瀏覽器
→
5
取得驗證結果

Step 1 — 輸入前端 URL

  • 輸入含 token 的遊戲進入 URL（含 gameid 等參數）
  • 格式：https://osm-redirect.osmslot.org/?token=xxx&gameid=xxx

Step 2 — 填入已刪除圖片清單

有三種方式輸入刪除清單：

  • 手動輸入：每行一個圖片路徑，如 assets/resources/jackpot/aruze1.png
  • 貼上 git diff 文字解析：點擊「📋 貼上 git diff 文字解析」，貼入 git diff 輸出，系統自動萃取刪除的圖片路徑（支援 git diff --name-status / git status / GUI 格式）
  • 截圖 AI 識別：點擊「🖼️ 上傳截圖 AI 識別」或直接 Ctrl+V 貼上截圖，Gemini 自動識別圖片路徑填入欄位

Step 3 — 啟動偵測

  • 點擊「🚀 開始偵測」，後端啟動無頭瀏覽器
  • 系統開始攔截所有圖片資源請求

Step 4 — 操作虛擬瀏覽器

  • 畫面顯示即時截圖（每 1.5 秒自動更新）
  • 可直接點擊截圖操作遊戲（點擊位置會傳送至後端執行）
  • 支援滾輪捲動
  • 左上方顯示已攔截圖片請求數量
  • 操作遊戲進入各個畫面（大廳、遊戲房間、各功能頁面）讓系統完整收集圖片請求

Step 5 — 取得驗證結果

  • 點擊「✅ 完成，取得結果」
  • 系統輸出比對報告：
    • ❌ 仍在載入 — 圖片仍被前端請求，需回報給 RD 確認
    • ✅ 已確認刪除 — 圖片未出現在任何請求中，驗證通過
  • 點擊「← 重新偵測」可重新開始
